Five principles of SOLID are below,  most of them are promoted by [Rober C. Martin](https://en.wikipedia.org/wiki/SOLID). Let me explain with simple examples. 
- Single-responsiblity principle
- Open-closed principle
- Liskov substitution principle
- Interface segregation principle
- Dependency Inversion Principle

Not a thumb rule, but applying above principles helps to maintain software for longer period of time with less cost.

## 1. Single-responsiblity principle

Make it simple to understand easily by others and easy to debug. So here the concept is to keep Objects to do one and only one thing.

##### Scenario  
Write objects where it mimics a notepad and methods to output the in json or pdf format. 

#### Do's

```java
public class Doc {
    private String content
    Doc(content) {
        this.content = content
    }
}
```

```java
// Report in seperate class
public class Report {

    private String content
    private String reportJson
    private String reportPdf
    
    
    Report(doc) { 
        this.doc = doc
        this.reportJson = json()
        this.report_pdf = pdf()
     }
    
    private void json() { 
        // json encoding code
        return 
    }
    
    private void pdf() { 
        // pdf encoding code
        return 
    }
}
```

### Don't

```java
public class Doc {
    private String content
    private String reportJson
    private String reportPdf
    
    Doc(content) {
        this.content = content
        this.reportJson = json()
        this.report_pdf = pdf()
    }
    
    private void json() { 
        // json encoding code
        return 
    }
    
    private void pdf() { 
        // pdf encoding code
        return 
    }
}
```

<span style="color:red"> Why ? </span> Since generating report in different format will change from time to time, keeping it in Doc class would not be good idea, instead, keep Doc class just for storing the content of the Doc. 

#### Notes
- There is nothing wrong fundamentally in above code, but if you follow the SOLID design then it will be more maintainable and less buggy. 
- Keep classes to have Single responsiblity. 
- <span style="color:red"> Caution: </span> The above example is just to illustrate the Principle of Single Responsibility. Not to show how to design actual notepad.

## 2. Open-closed Principle

“Software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification.”

<b>Scenario:</b> Imagine you are building a MS Office word 2020, call it as **MSWord2020**. The above code is very old version, has just two types of export options(pdf, json), but new MSWord2020 has some new features:
    - such as text to speech. 
    - etc.
Also keep in mind, to add one more feature in MSWord2030 version. How will do you design you Objects ?

#### Do's

```java
interface doc 
{ 
	void display(String content); 
} 

interface doc_feature1 
{ 
	void feature1(String content); 
} 

class MSWord2020 implements doc, doc_feature1
{ 
    
	public void display(String content) 
	{  ... } 
    
    public void feature1(String content) 
	{  ... } 

}

```

**Add new feature in 2030:**

Now think how to deal with new MSWord2030 version, if you think you will provide all features of MSWord2020, then it is better to extend the MSWord2020, then implement new interface features. Something like below


```java
interface doc_feature2
{
    void feature2(String content);
}
class MSWord2030 extends MSWord2020 implements doc_feature2, 
{ 
    public void feature2(String content) 
    {  ... } 

}
```

#### Don't 

<b> you can modify the original class to add new feature 2, like below</b> 
```java
class Doc {
    void display(String content) {}
    void feature1(String content) {}
    void feature3(String content) {}
}
```

<p style="color:red"> Caution:  This code does not obey Rule 1, i.e, single reponsiblity principle </p>


#### Notes:
- Take a look at the don't part, you see you might need to modify the original class. So <b> avoid it. </b>. 
- Its always better to program your code in Interfaces and apply inheritance where possible.

## 3. Liskov substitution principle

This is most import and tricky rule to understand, I read several blogs and guides. Try to read this one atleast twice.

#### Pre-requirement for this rule
- Are you implementing an interface ? If Yes, you must test this rule throughly. 

**Rule:**
Any class, S, can be used to replace a class, B, if and only if, S is a subtype of B. 

```java
public interface Human {
    publi void eat();
}
public class HumanoidRobot extends Human {
     // can robot eat ? 
}
    
Human obj1 = new Human()
Human obj2 = new Robot() 
```

If you apply the rule S is Human and B is Robot, suppose if you replace B with S(meaning, HumanoidRobot with Human) then it does not make sense to many operations, such as eating method of human. So one way to avoid is to like below..

**Don't**

```java
public interface Human {
    publi void eat();
}
public class HumanoidRobot extends Human {
    // Humanoid Robot  - A humanoid is something that has an appearance 
    // resembling a human without actually being one.
    // can robot eat ? No
    @override
    publi void eat() {
        // do nothing
    }
}
```

**Liskov Substitution Principle** states that:

If a class, S, is a subtype of a class, B, then S can be used to replace all instances of B
without changing the behaviors of a program. The logic behind this is straightforward. If S is a subtype of B, then it can be expected that S will have the same behaviors as B. Therefore, S can be used in place of B and it would not affect the software. This means that inheritance can be tested by applying substitution.

**Solution**
```java
public interface Everything { run() } 
// similar to Object class(root class, it has most generalized methods) in Java 

public class HumanoidRobot extends Everything { 
    public void run() { 
        // code to run using batteries.
    }
}
public class Human extends Everything {
    public void run() { 
        // code to run using physical streangth.
    }
}

// Example:
    Everything ev = new HumanoidRobot()
    Everything ev = new Human()
// if you apply the substition rule it satisfies, since both does the run operation but differently. 
```

**Notes**:

- [The base class is the more generalized class, and therefore, its attributes and behaviors should reflect it. The names given to the attributes and methods, as well as the implementation of each method must be broad enough that all subclasses can use them.](https://www.coursera.org/learn/design-patterns)

- [If inheritance is not used correctly, it can lead to a violation of the “Liskov Substitution Principle”. This principle uses substitution to determine whether or not inheritance has been properly used.](https://www.coursera.org/learn/design-patterns)

- [These rules are not programmatically enforced by any object-oriented language. In fact, overriding a base class's behaviors can have advantages. Subclasses can improve the performance of behaviours of its base class, without changing the expected results of said behavior.](https://www.coursera.org/learn/design-patterns)
- **Another example:**  [let's take a look at a class that is an abstraction of a department store. The base class may implement a naive searching algorithm that, in the worst case, iterates through the entire list of the items that the store sells. A subclass could override this method and provide a better search algorithm. Although the approach that the subclass takes to searching is different, the expected behavior and outcome is the same.](https://www.coursera.org/learn/design-patterns)

# 4.  Interface Segregation Principle

Suppose you have an interface for a Robot, operations include run, fight, walk, swim, shoot. Now another company wants to use your code base for Robot and extend where the Robot can Speak. But here is the problem the company don't want to implement swim, fight and shoot operations. So company writes something like below - 

### **Don't** 
```java

public interface Robot {
     void run();
     void shoot();
     void swim();
     void fight();
     void walk();
 }

// company A wants to extend the Robot to add Speaking feature


public SpeakingRobot extends Robot {
     @Overrides 
     void run() {
         // .. inherits from Robot
     }
    
     @Overrides 
     void shoot() {
         // do nothing 
         // return;
     }
    
     @Overrides 
     void swim() {
          // do nothing 
         // return;
     }
    
     @Overrides 
     void fight(){
          // do nothing 
         // return;
     }
    
     @Overrides 
     void walk() {
         // .. inherits from Robot
     }
    
     @Overrides 
     void speak() {
        // code to speak
     }
 }


```

What is the problem? 
- You are making client or company to implement the operations forcefully. So in order avoid this we need to have interface segregation principle. Its quite simple, that **no class should be forced to depend on methods it does not use.**

### Do

```java
interface Robot {
    // very common features, every client must implement are below.
    void run();
    void walk();
}

interface ShootingRobot extends Robot { void shoot(); }
interface SwimmingRobot extends Robot { void swim(); }
interface FightingRobot extends Robot { void fight(); }

class  SpeakingRobot  implements  Robot{


    void speak() {
        // code to speak
    }


    @Override
    public void run() {
        // code to run 
    }

    @Override
    public void walk() {
        // code to walk
    }
}

```

# 5. Dependency Inversion Principle 

Wikipedia states the principle as:

- High-level modules should not depend on low-level modules. Both should depend on abstractions.
- Abstractions should not depend on details. Details should depend on abstractions.

Quite easy to understand using examples 

First, Take a look at [Java Object Class: Root class to all objects](https://docs.oracle.com/javase/7/docs/api/index.html?java/lang/Object.html). For example, Take Number, String, Double etc Objects in java extends Object class. 

All above principles apply to this one. Here suppose you have Robot interface, then all objects of Robot should be referring to an abstraction rather than the concrete implementation. Referring to the above code:

####  Do's
```java
Robot r = new SpeakingRobot()
```

#### Don't
```java
SpeakingRobot r = new SpeakingRobot() 
```

SpeakingRobot class is an actual implementation(low-level class), so any object which might change in future should depend on abstractions(Robot interface is high-level abstraction here).
